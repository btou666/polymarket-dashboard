# Polymarket 流动性奖励看板 PRD（优化版）

版本：V1.0  
状态：可用于产品评审 / 设计出稿 / 技术评估  
更新时间：2026-03-18

## 1. 项目背景

Polymarket 的流动性奖励机制会对做市用户的挂单行为进行奖励，但用户在参与前通常无法快速判断：

- 当前哪些奖励池更值得投入资金
- 相同资金在不同池子中的预估小时收益差异
- 单边挂单和双边挂单的收益效率差异
- 靠近 midpoint 的更高奖励是否值得承担更高成交风险

现状是，用户往往需要先挂单，再观察平台侧的奖励表现，决策链路长、试错成本高、资金配置效率低。

因此需要一个面向做市用户的决策看板，在挂单前基于当前市场快照和奖励规则，对不同池子的预估收益效率做可视化比较。

## 2. 产品目标

### 2.1 核心目标

帮助用户在挂单前回答以下问题：

- 我现在应该优先投哪个奖励池
- 如果投入 1000 USDC，不同池子的预估小时收益分别是多少
- 当前哪些池子属于“奖励高、竞争相对低”
- 在某个池子中，单边挂单和双边挂单的预估收益差多少

### 2.2 业务价值

对用户：

- 降低试错成本
- 提升资金配置效率
- 提升做市参与意愿

对平台或产品方：

- 提升奖励机制透明度
- 促进流动性进入更有效的市场
- 为后续策略推荐与自动化工具提供基础数据层

## 3. 产品定位

一句话定义：

输入金额，输出各奖励池的预估小时奖励、单位资金效率和竞争情况，帮助用户做挂单前决策。

本产品的定位是“挂单前预估工具”，不是“实际收益结算工具”。

## 4. 目标用户

### 4.1 目标用户

- Polymarket 活跃做市用户
- 高频挂单 / 撤单交易者
- 奖励敏感型套利或量化用户
- 需要横向比较多个奖励池的流动性提供者

### 4.2 用户特征

- 理解订单簿和盘口竞争
- 更关注资金效率而非名义奖励金额
- 接受“基于当前快照的估算值”，但不接受黑盒推荐
- 需要先看到结果，再按需查看计算逻辑

## 5. 核心问题与非目标

### 5.1 要解决的问题

将复杂的奖励规则、盘口竞争情况和用户拟投入金额，转化为用户可直接比较的几个结果指标：

- 预估每小时奖励
- 每 1000 USDC 预估收益
- ROI/h
- 竞争强度

### 5.2 本期非目标

本期不解决以下问题：

- 自动下单
- 自动调仓或跨池资金分配
- 账户级真实收益归因
- 历史交易收益分析
- 风险对冲建议
- 保证收益或替代用户做交易判断

## 6. MVP 范围

### 6.1 MVP 目标

验证用户是否会基于该看板调整流动性奖励池选择，并愿意在挂单前进行收益模拟。

### 6.2 MVP 只做

- 奖励池总览页
- 统一金额输入
- 基于当前市场快照的预估每小时奖励计算
- 每 1000 USDC 收益计算
- 竞争强度展示
- 基础排序与筛选
- 单池详情页
- 单池基础策略对比：单边 / 双边、距 midpoint 档位

### 6.3 MVP 暂不做

- 竞争结构分析独立页面
- 24h / 7d 趋势分析
- 多档拆单模拟
- 高级模式输入项，如撤单频率、在线率、被成交概率
- 历史回测
- 个性化推荐策略
- 自动报价建议
- 账户资产打通

## 7. 用户决策链路

核心路径如下：

1. 用户进入总览页，输入统一模拟金额
2. 用户按预估小时奖励、ROI/h 或竞争强度排序
3. 用户筛出高效率池子
4. 用户进入单池详情页，对比单边 / 双边和不同报价距离的预估结果
5. 用户跳转到目标市场执行挂单

产品应满足以下体验目标：

- 3 秒内看出当前最值得关注的池子
- 10 秒内完成一次单池基础测算
- 默认先展示结果，公式和解释按需展开

## 8. 核心指标定义

### 8.1 每小时总奖励

每小时总奖励 = 每日总奖励 / 24

### 8.2 用户投入金额

用户计划投入到当前市场的总资金金额，单位为 USDC。

为避免口径歧义，MVP 统一定义为：

- 单边挂单：输入金额即单侧投入金额
- 双边挂单：输入金额为总投入金额，系统默认平均分配到双侧

### 8.3 用户有效挂单金额

满足平台奖励规则，且位于有效价差范围内、满足最小挂单要求的金额。

### 8.4 用户预估有效分数

基于用户输入的金额、挂单方式、报价距 midpoint 的距离，按奖励规则估算出的用户得分。

### 8.5 市场当前总有效分数

基于当前订单簿快照估算出的市场总有效分数。

### 8.6 预估奖励占比

预估奖励占比 = 用户预估有效分数 / （市场当前总有效分数 + 用户预估有效分数）

### 8.7 预估每小时奖励

预估每小时奖励 = 每小时总奖励 x 预估奖励占比

### 8.8 单位资金小时收益

单位资金小时收益 = 预估每小时奖励 / 用户投入金额

### 8.9 ROI/h

ROI/h = 单位资金小时收益，以百分比展示。

### 8.10 竞争强度

竞争强度用于描述当前池子是否拥挤。MVP 中以 0-100 的内部评分计算，并映射为：

- 0-33：低
- 34-66：中
- 67-100：高

评分输入建议包括：

- 当前有效挂单总量
- 最优价差附近挂单密度
- 双边有效流动性占比
- 头部档位集中度

## 9. 预估模型设计

### 9.1 模型原则

MVP 采用“当前快照预估”而不是“历史趋势预测”。

模型目标不是拟合实际结算结果到小数点后，而是帮助用户做相对比较和初步筛选。

### 9.2 模型输入

用户输入：

- 投入金额
- 挂单方式：单边 / 双边
- 挂单方向：买 / 卖 / 双边
- 报价距离 midpoint 的档位
- 挂单时长：1h / 4h / 24h

系统输入：

- 每日奖励金额
- 市场参数，如 max spread、min size
- 当前 midpoint
- 当前订单簿快照
- 当前市场状态

### 9.3 模型输出

- 用户预估有效分数
- 市场当前总有效分数
- 预估奖励占比
- 预估每小时奖励
- 预估每日奖励
- 每 1000 USDC 收益
- ROI/h

### 9.4 模型展示策略

为了避免“伪精确”，MVP 默认展示中性预估，并向用户明确说明：

- 结果基于当前市场快照
- 结果用于相对比较，不代表实际结算结果
- 市场竞争和订单成交会使实际收益发生变化

MVP 不强制展示保守 / 中性 / 激进三档。该能力放入 V2，前提是模型稳定性和用户理解成本可接受。

### 9.5 模型待确认项

以下内容需在开发前与实际奖励规则再次校验：

- 单边与双边得分差异的具体权重
- midpoint 距离对得分的衰减规则
- 订单有效性边界条件
- 奖励采样周期与结算周期的口径

若官方规则存在口径变动，产品展示应优先与最新规则保持一致。

## 10. 数据需求

### 10.1 外部数据

需要获取：

- 奖励市场列表
- 每个市场奖励额度
- 市场参数，如 max spread、min size
- 当前订单簿快照
- midpoint
- bid / ask 深度
- 市场状态

### 10.2 系统内部计算数据

系统需生成：

- 每个市场的当前总有效分数
- 用户模拟挂单后的新增有效分数
- 用户加入后的预估奖励占比
- 预估每小时奖励和单位资金收益
- 竞争强度评分

### 10.3 数据刷新要求

MVP 建议采用准实时刷新：

- 总览页自动刷新间隔：30-60 秒
- 单池详情页支持手动刷新
- 页面显式展示“最近更新时间”

### 10.4 数据异常策略

若发生以下情况，页面需明确提示而不是继续给出看似准确的结果：

- 订单簿数据缺失
- midpoint 缺失
- 市场状态异常
- 数据更新时间超过阈值

异常时对应池子展示：

- `暂不可估算`
- `数据延迟`
- `市场状态异常`

## 11. 功能需求

### 11.1 页面一：奖励池总览页

### 功能目标

让用户快速横向比较所有奖励池的资金效率，筛出当前值得进一步查看的市场。

### 展示字段

- 市场名称
- 市场状态
- 每日总奖励
- 每小时总奖励
- max spread
- min size
- 当前 midpoint
- 竞争强度
- 输入金额下的预估每小时奖励
- 预估每日奖励
- 每 1000 USDC 收益
- ROI/h
- 推荐标签
- 数据更新时间

### 页面交互

- 顶部统一输入模拟金额，如 100 / 500 / 1000 / 5000 USDC
- 支持排序：
  - 按预估每小时奖励
  - 按 ROI/h
  - 按竞争强度
  - 按总奖励
- 支持筛选：
  - 高奖励
  - 低竞争
  - 适合双边
- 支持按市场名称搜索

### 推荐标签规则

MVP 中先使用简单规则生成标签：

- `适合双边`：双边模拟收益显著高于单边
- `竞争激烈`：竞争强度为高
- `高奖励`：每小时总奖励位于前 25%
- `低竞争`：竞争强度为低

### 验收标准

- 用户输入金额后，列表结果在可接受时间内刷新
- 排序和筛选可组合使用
- 数据异常的池子不能展示正常收益值
- 用户能从总览页点击进入单池详情页

### 11.2 页面二：单池详情页

### 功能目标

让用户针对目标池子做基础策略对比，理解收益差异来自哪里。

### 输入项

- 投入金额
- 挂单方式：单边 / 双边
- 挂单方向：买 / 卖 / 双边
- 报价距离 midpoint 的档位
- 挂单时长：1h / 4h / 24h

### 输出项

- 用户预估有效分数
- 市场当前总有效分数
- 预估奖励占比
- 预估每小时奖励
- 预估每日奖励
- 每 1000 USDC 收益
- ROI/h
- 风险提示
- 数据更新时间

### 图表

MVP 只保留两个最关键图表：

- 图 1：投入金额 vs 预估每小时奖励
- 图 2：报价距离 midpoint 档位 vs 预估每小时奖励

“你加入前后奖励占比变化”放入 V2。

### 验收标准

- 用户修改任一输入项后，结果可重新计算
- 图表与数值结果口径一致
- 单边与双边的资金口径一致且说明清晰
- 若当前池子数据缺失，则整页给出不可估算提示

## 12. 交互与体验要求

### 12.1 信息层级

- 第一层：直接给决策结果
- 第二层：给指标解释
- 第三层：给公式和模型说明

### 12.2 展示原则

- 总览页偏决策
- 单池详情页偏解释
- 不在 MVP 内单独做深度分析页

### 12.3 风险提示

每个结果区域都需固定展示以下提示：

- 结果基于当前盘口快照估算，不代表实际结算结果
- 越接近 midpoint 的报价通常奖励更高，但成交风险也更高
- 市场竞争变化可能导致实际收益快速变化

建议同时展示一个轻量级“置信提示”：

- 数据新鲜
- 数据一般
- 数据延迟

## 13. 埋点与成功指标

### 13.1 用户行为埋点

建议至少埋以下事件：

- 进入总览页
- 输入模拟金额
- 修改排序 / 筛选条件
- 点击进入单池详情
- 在单池页切换单边 / 双边
- 在单池页修改报价距离档位
- 从单池页跳转市场页

### 13.2 MVP 成功指标

优先使用可直接观测的指标：

- 总览页访问人数
- 人均查看池子数
- 模拟金额输入率
- 单池详情点击率
- 单池模拟完成率
- 从看板跳转市场页的转化率
- 7 日重复访问率

### 13.3 暂不作为 MVP 核心指标

以下指标暂不建议作为一期核心目标，因为难以可靠归因：

- 用户基于看板调整池子选择的比例
- 单用户奖励资金效率提升
- 长尾池流动性改善

这些指标可以在后续接入账户或订单级数据后再评估。

## 14. 风险与应对

### 14.1 模型精度风险

风险：

- 预估值与实际结算结果存在偏差

应对：

- 强调结果用于比较而非承诺
- 避免展示过度精确的小数
- 用风险提示和数据更新时间帮助用户正确理解

### 14.2 数据实时性风险

风险：

- 订单簿变化快，旧快照会误导用户

应对：

- 显示最近更新时间
- 单池详情支持手动刷新
- 超时数据降级展示为 `数据延迟`

### 14.3 用户误解风险

风险：

- 用户将高 ROI 直接理解为低风险高收益

应对：

- 在结果区固定展示成交风险提示
- 不使用“保证收益”“将获得”等表述
- 推荐标签避免绝对化措辞

## 15. 开发前待确认事项

以下事项建议在进入设计或开发前完成确认：

1. 奖励规则的最新口径和核心公式
2. 可用的数据接口、字段完整性和限频要求
3. 总览页与详情页的目标刷新频率
4. 竞争强度评分的第一版计算方式
5. 是否支持市场页跳转以及跳转目标地址规则

## 16. 版本路线图

### V1

- 总览页
- 单池详情页
- 快照式收益预估
- 基础排序筛选

### V2

- 保守 / 中性 / 激进三档预估
- 竞争结构深度分析
- 加入前后占比变化
- 历史快照趋势

### V3

- 接入用户真实挂单数据
- 展示“我的挂单收益预估”
- 低效资金占用识别

### V4

- 跨池资金分配建议
- 自动策略推荐
- 奖励收益与成交 / 库存风险联合优化

## 17. 一句话同步版本

这是一个帮助 Polymarket 做市用户在挂单前比较各奖励池预估小时收益和资金效率的决策看板，MVP 聚焦“快照式横向比较 + 单池基础模拟”，目标是提升流动性奖励的资金配置效率。
